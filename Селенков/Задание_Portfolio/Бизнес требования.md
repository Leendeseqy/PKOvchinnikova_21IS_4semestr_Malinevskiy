# Бизнес-требования к приложению "Portfolio Manager"

## 1. Введение

### 1.1. Цель документа
Настоящий документ описывает бизнес-требования к приложению "Portfolio Manager" - системе управления портфолио научных, профессиональных и творческих достижений.

### 1.2. Область действия
Документ охватывает функциональные и нефункциональные требования, ограничения, бизнес-правила и сценарии использования приложения.

### 1.3. Целевая аудитория
- Руководители проектов
- Разработчики
- Тестировщики
- Научные сотрудники
- Преподаватели
- Специалисты, ведущие портфолио достижений

### 1.4. Бизнес-цели
1. Создать централизованную систему управления профессиональными достижениями
2. Упростить процесс документирования и каталогизации работ
3. Автоматизировать генерацию отчетов и аналитики
4. Обеспечить возможность экспорта данных в стандартные форматы
5. Поддержка совместной работы (соавторство)

## 2. Общее описание

### 2.1. Обзор продукта
Portfolio Manager - десктопное приложение на Python с графическим интерфейсом (Tkinter) для управления коллекцией профессиональных записей (статьи, книги, проекты, доклады, патенты).

### 2.2. Функциональные возможности
- Создание, редактирование, удаление записей портфолио
- Хранение описаний в формате Markdown
- Управление соавторами
- Сортировка и фильтрация записей
- Экспорт данных в Excel и Word
- Генерация аналитических отчетов с графиками
- Просмотр записей в HTML-формате

### 2.3. Ограничения
- Приложение работает только на Windows/Linux/macOS с Python 3.8+
- Требуется установка зависимостей через pip
- Локальное хранение данных (SQLite)
- Отсутствие сетевого доступа к данным
- Максимальный размер файлов: 10 МБ на запись

## 3. Функциональные требования

### 3.1. Управление записями портфолио

#### FR-001: Создание записи
**Приоритет:** Высокий
**Описание:** Пользователь должен иметь возможность создавать новые записи в портфолио
**Критерии приемки:**
- Поля: название, тип, год (обязательные)
- Автоматическое создание Markdown-файла
- Присвоение уникального ID
- Запись в базу данных SQLite
- Создание лога активности

#### FR-002: Редактирование записи
**Приоритет:** Высокий
**Описание:** Пользователь должен иметь возможность редактировать существующие записи
**Критерии приемки:**
- Обновление основных полей (название, тип, год)
- Редактирование Markdown-содержимого
- Автоматическое обновление имени файла при изменении названия
- Обновление timestamp в БД

#### FR-003: Удаление записи
**Приоритет:** Высокий
**Описание:** Пользователь должен иметь возможность удалять записи
**Критерии приемки:**
- Подтверждение перед удалением
- Удаление записи из БД
- Удаление связанного Markdown-файла
- Каскадное удаление соавторов
- Сохранение лога удаления

#### FR-004: Просмотр записи
**Приоритет:** Средний
**Описание:** Пользователь должен иметь возможность просматривать записи
**Критерии приемки:**
- Отображение в текстовом редакторе
- HTML-превью в браузере
- Подсветка синтаксиса Markdown
- Отображение метаданных

### 3.2. Управление соавторами

#### FR-005: Добавление соавтора
**Приоритет:** Средний
**Описание:** Пользователь должен иметь возможность добавлять соавторов к записям
**Критерии приемки:**
- Ввод имени соавтора
- Проверка на уникальность (в рамках записи)
- Отображение в списке записи
- Хранение в отдельной таблице БД

#### FR-006: Удаление соавтора
**Приоритет:** Низкий
**Описание:** Пользователь должен иметь возможность удалять соавторов
**Критерии приемки:**
- Удаление из списка соавторов
- Обновление отображения
- Удаление из БД

### 3.3. Сортировка и фильтрация

#### FR-007: Сортировка записей
**Приоритет:** Средний
**Описание:** Пользователь должен иметь возможность сортировать записи по различным критериям
**Критерии приемки:**
- Сортировка по ID, названию, типу, году, дате создания
- Переключаемое направление сортировки (возрастание/убывание)
- Визуальная индикация активной сортировки

### 3.4. Экспорт данных

#### FR-008: Экспорт в Excel
**Приоритет:** Высокий
**Описание:** Пользователь должен иметь возможность экспортировать данные в Excel
**Критерии приемки:**
- Экспорт всех записей
- Сохранение структуры данных
- Настраиваемое имя файла
- Корректное отображение кириллицы
- Форматирование столбцов

#### FR-009: Экспорт в Word
**Приоритет:** Высокий
**Описание:** Пользователь должен иметь возможность экспортировать данные в Word
**Критерии приемки:**
- Экспорт с сохранением форматирования
- Обработка Markdown-разметки
- Создание структурированного документа
- Титульная страница
- Разделение записей

#### FR-010: Генерация отчетов
**Приоритет:** Средний
**Описание:** Система должна генерировать аналитические отчеты
**Критерии приемки:**
- Статистика по типам записей
- Распределение по годам
- Активность пользователя
- Графики в Excel-отчете
- Полный список записей в Word-отчете

### 3.5. Аналитика и статистика

#### FR-011: Сбор статистики
**Приоритет:** Низкий
**Описание:** Система должна собирать статистику использования
**Критерии приемки:**
- Количество записей по типам
- Распределение по годам
- Количество уникальных соавторов
- Активность за период
- Последние записи

## 4. Нефункциональные требования

### 4.1. Требования к производительности

#### NFR-001: Время отклика
**Описание:** Время отклика интерфейса не должно превышать 1 секунду
**Метрики:**
- Загрузка списка записей: < 2 сек
- Открытие записи: < 1 сек
- Генерация отчета: < 30 сек
- Экспорт данных: < 10 сек

#### NFR-002: Использование ресурсов
**Описание:** Приложение должно эффективно использовать ресурсы
**Метрики:**
- Потребление памяти: < 200 МБ
- Использование CPU: < 20% в idle
- Размер БД: эффективное хранение

### 4.2. Требования к надежности

#### NFR-003: Доступность
**Описание:** Приложение должно быть доступно для работы 99% времени
**Метрики:**
- Отсутствие критических сбоев
- Восстановление после ошибок
- Сохранность данных при аварийном завершении

#### NFR-004: Целостность данных
**Описание:** Обеспечение целостности и консистентности данных
**Метрики:**
- Транзакционность операций с БД
- Проверка целостности связей
- Резервное копирование не требуется (локальное приложение)

### 4.3. Требования к удобству использования

#### NFR-005: Интерфейс пользователя
**Описание:** Интуитивно понятный и удобный интерфейс
**Критерии:**
- Русскоязычный интерфейс
- Логичная группировка элементов
- Подсказки и сообщения об ошибках
- Соответствие принципам GUI

#### NFR-006: Обучение
**Описание:** Минимальное время обучения работе с приложением
**Критерии:**
- Опытный пользователь осваивает за 30 минут
- Наличие подсказок в интерфейсе
- Понятные названия функций

### 4.4. Технические требования

#### NFR-007: Совместимость
**Описание:** Совместимость с различными ОС и версиями Python
**Критерии:**
- Поддержка Windows 10+, Ubuntu 18.04+, macOS 10.15+
- Поддержка Python 3.8+
- Кодировка UTF-8

#### NFR-008: Безопасность
**Описание:** Базовые требования безопасности
**Критерии:**
- Локальное хранение данных
- Нет требований к аутентификации
- Валидация входных данных

## 5. Бизнес-правила

### 5.1. Правила валидации данных

#### BR-001: Валидация года
**Описание:** Год записи должен быть корректным
**Правило:**
- Год должен состоять из 4 цифр
- Допустимый диапазон: 1900-текущий год+5
- Только целые числа

#### BR-002: Валидация названия
**Описание:** Название записи должно соответствовать правилам
**Правило:**
- Не пустая строка
- Максимальная длина: 200 символов
- Допустимые символы: буквы, цифры, пробелы, дефисы, подчеркивания

#### BR-003: Уникальность соавторов
**Описание:** Соавторы не должны повторяться в рамках одной записи
**Правило:**
- Имя соавтора должно быть уникальным для каждой записи
- Регистрозависимое сравнение
- Удаление лишних пробелов

### 5.2. Правила именования файлов

#### BR-004: Генерация имени файла
**Описание:** Автоматическая генерация имен файлов
**Правило:**
- Формат: `Название_Год.md`
- Замена пробелов на подчеркивания
- Удаление недопустимых символов
- Сохранение в папке `records`

### 5.3. Правила ведения логов

#### BR-005: Логирование действий
**Описание:** Все значимые действия должны логироваться
**Правило:**
- Логирование создания, изменения, удаления записей
- Хранение в таблице `activity_log`
- Формат: timestamp, action, details

## 6. Сценарии использования

### 6.1. Основные сценарии

#### UC-001: Создание новой записи портфолио
**Актор:** Пользователь
**Предусловие:** Приложение запущено
**Основной поток:**
1. Пользователь вводит название, выбирает тип, указывает год
2. Нажимает кнопку "Создать"
3. Система валидирует данные
4. Система создает запись в БД и Markdown-файл
5. Система добавляет запись в список
6. Система отображает сообщение об успехе

**Альтернативный поток A:** Неверные данные
**Альтернативный поток B:** Ошибка создания файла

#### UC-002: Экспорт портфолио в Word
**Актор:** Пользователь
**Предусловие:** Есть хотя бы одна запись
**Основной поток:**
1. Пользователь нажимает "Экспорт в Word"
2. Система запрашивает путь для сохранения
3. Пользователь выбирает путь и имя файла
4. Система генерирует Word-документ
5. Система обрабатывает Markdown-разметку
6. Система сохраняет файл
7. Система уведомляет об успешном экспорте

**Альтернативный поток:** Нет записей для экспорта

### 6.2. Сценарии отчетности

#### UC-003: Генерация аналитического отчета
**Актор:** Пользователь
**Предусловие:** Приложение содержит данные
**Основной поток:**
1. Пользователь нажимает "Сформировать отчет"
2. Система собирает статистику
3. Система генерирует Excel-файл с графиками
4. Система генерирует Word-файл с подробными данными
5. Система сохраняет файлы в папку `reports`
6. Система уведомляет о завершении

## 7. Требования к интерфейсу

### 7.1. Структура интерфейса
- Левая панель: управление и список записей
- Правая панель: редактирование и дополнительные функции
- Статус-бар: информация о текущем состоянии

### 7.2. Элементы управления
- Treeview для отображения списка записей
- ScrolledText для редактирования Markdown
- Combobox для выбора типа записи
- Кнопки с понятными надписями
- Поля ввода с валидацией

### 7.3. Визуальное оформление
- Четкое разделение функциональных блоков
- Контрастные цвета для важных элементов
- Единый стиль шрифтов
- Индикация выбранных элементов

## 8. Ограничения и допущения

### 8.1. Ограничения
- Нет облачной синхронизации
- Нет многопользовательского режима
- Нет API для интеграции
- Нет мобильной версии
- Офлайн-работа только

### 8.2. Допущения
- Пользователь имеет базовые знания Markdown
- Данные не являются конфиденциальными
- Объем данных не превышает 10,000 записей
- Пользователь работает на персональном компьютере
