# План тестирования приложения "Portfolio Manager"

## 1. Введение

### 1.1. Цель документа
Настоящий документ описывает стратегию, объем, подход и расписание тестирования десктопного приложения "Portfolio Manager". План предназначен для координации деятельности по тестированию и обеспечения качества продукта.

### 1.2. Область тестирования
Тестирование охватывает все функциональные и нефункциональные требования, описанные в бизнес-требованиях, включая:
- Функциональность управления записями
- Работу с базой данных
- Экспорт данных
- Генерацию отчетов
- Пользовательский интерфейс
- Производительность и надежность

### 1.3. Целевая аудитория
- Тестировщики (QA Engineers)
- Разработчики
- Руководитель проекта
- Бизнес-аналитик

## 2. Цели и задачи тестирования

### 2.1. Основные цели
1. Подтверждение соответствия приложения бизнес-требованиям
2. Выявление дефектов на ранних этапах разработки
3. Оценка качества и стабильности приложения
4. Обеспечение корректной работы всех функций

### 2.2. Задачи тестирования
- Провести функциональное тестирование всех модулей
- Выполнить тестирование интерфейса (GUI)
- Проверить корректность работы с базой данных
- Протестировать экспорт данных в различные форматы
- Оценить производительность приложения
- Провести интеграционное тестирование
- Выполнить регрессионное тестирование

## 3. Подход и стратегия тестирования

### 3.1. Методологии тестирования
- **Модульное тестирование:** PyTest для тестирования отдельных функций и классов
- **Интеграционное тестирование:** Проверка взаимодействия модулей
- **Системное тестирование:** Тестирование приложения как целостной системы
- **Приемочное тестирование:** Проверка соответствия бизнес-требованиям

### 3.2. Типы тестирования
| Тип тестирования | Методы | Инструменты |
|-----------------|--------|-------------|
| Функциональное | Black-box, тестирование граничных значений, эквивалентных классов | PyTest, ручное тестирование |
| GUI тестирование | Тестирование макета, навигации, состояний элементов | Selenium (опционально), ручное тестирование |
| Тестирование базы данных | Проверка целостности данных, транзакций, запросов | SQLite командная строка, PyTest |
| Тестирование производительности | Замер времени отклика, использование ресурсов | time, memory_profiler |
| Тестирование безопасности | Валидация входных данных, SQL-инъекции | PyTest, специальные скрипты |

### 3.3. Критерии начала тестирования
- Завершена разработка всех основных функций
- Создана тестовая среда
- Подготовлены тестовые данные
- Утверждены тест-кейсы

### 3.4. Критерии завершения тестирования
- Выполнены все запланированные тесты
- Исправлены все критические и блокирующие дефекты
- Уровень покрытия тестами ≥ 80%
- Успешно пройдено приемочное тестирование

### 8.2. Формат тест-кейсов
Каждый тест-кейс должен содержать:

- Статус (Pass/Fail/Blocked)
- Приоритет (High/Medium/Low)
- Связанное требование

## 9. Процесс тестирования

### 9.1. Жизненный цикл дефекта
1. **Обнаружение:** Тестировщик обнаруживает дефект
2. **Документирование:** Создание issue в системе отслеживания
3. **Приоритизация:** Назначение приоритета (Blocker/Critical/Major/Minor)
4. **Назначение:** Назначение разработчику
5. **Исправление:** Разработчик исправляет дефект
6. **Повторное тестирование:** Проверка исправления
7. **Закрытие:** Дефект закрывается после подтверждения исправления

### 9.2. Критерии приоритизации дефектов
| Приоритет | Описание | Примеры |
|-----------|----------|---------|
| Blocker | Полностью блокирует работу приложения | Критическая ошибка при запуске |
| Critical | Важная функция не работает | Невозможно создать запись |
| Major | Функция работает с ограничениями | Экспорт без форматирования |
| Minor | Косметические проблемы | Неверное выравнивание текста |

### 9.3. Процесс отчетности
- **Ежедневные отчеты:** Краткий статус выполнения тестов
- **Недельные отчеты:** Детальный отчет о прогрессе и дефектах
- **Финальный отчет:** Полный отчет по результатам тестирования

## 10. Критерии приемки

### 10.1. Функциональные критерии
- Все функции из бизнес-требований реализованы и работают корректно
- Успешно пройдены все тест-кейсы высокого приоритета
- Уровень успешных тестов ≥ 95%
- Все критичные дефекты исправлены

### 10.2. Нефункциональные критерии
- Время отклика соответствует требованиям
- Приложение стабильно работает в течение 24 часов
- Потребление ресурсов в пределах нормы
- Корректная работа на поддерживаемых ОС

### 10.3. Критерии качества кода
- Покрытие тестами ≥ 80%
- Отсутствие утечек памяти
- Корректная обработка исключений
- Соответствие PEP 8

## 11. Автоматизация тестирования

### 11.1. Области автоматизации
- Модульное тестирование классов и функций
- Тестирование работы с базой данных
- Тестирование экспорта данных
- Тестирование валидации входных данных

### 11.2. Инструменты автоматизации
- PyTest для модульного тестирования
- pytest-mock для мокирования
- coverage для измерения покрытия
- tox для тестирования в разных окружениях

### 11.3. Стратегия автоматизации
1. Начать с модульного тестирования бизнес-логики
2. Добавить интеграционные тесты
3. Реализовать тесты для критичных функций
4. Постепенно увеличивать покрытие

## 12. Матрица соответствия требований
